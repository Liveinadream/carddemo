# 分段式卡牌探索地图设计方案

基于当前项目的卡牌架构，我们设计一套支持**分段加载**、**自由移动**且**事件高度卡牌化**的地图系统。

---

## 1. 核心设计概念：分段加载 (Segmented Loading)

为了节省内存并提高加载速度，我们将整个大地图拆分为多个独立的 `MapSegment`（地图片段）。

### **1.1 数据结构**
每个 `MapSegment` 存储在一个独立的 JSON 文件或 Godot 资源文件中。
- **文件示例**: `res://Data/Maps/map_segment_1.json`
- **包含内容**:
  - `SegmentID`: 唯一标识。
  - `Layout`: 节点的坐标、类型（战斗、商店、随机、出口）。
  - `Connections`: 节点间的路径连通关系。
  - `Background`: 当前片段使用的背景图或环境。

### **1.2 加载逻辑 (MapManager)**
`MapManager` 仅实例化当前玩家所在的片段。
- **切换流程**: 
  1. 玩家移动到“出口”节点。
  2. `MapManager` 释放当前片段资源。
  3. 异步加载下一个片段的 JSON 数据并重建地图 UI。
- **代码思路**:
```csharp
public void LoadMapSegment(int segmentId) {
    // 释放旧节点
    ClearCurrentNodes();
    // 从 JSON 加载数据
    var data = StorageManager.Instance.LoadData<MapData>($"map_segment_{segmentId}");
    // 实例化新的地图 UI
    CreateMapUI(data);
}
```

## 2. 地图移动与探索 (Exploration)

地图呈现为一个“节点网格”或“卡牌棋盘”。

- **四处走动**: 玩家在地图上有一个代表自己的“头像”或“卡片标记”。
- **移动规则**: 玩家可以点击相邻的已连接节点进行移动。
- **战争迷雾**: 仅显示玩家当前节点及相邻 1-2 步范围内的节点。

## 3. 多卡牌场景交互事件 (Card-Based Events)

这是本项目最核心的特色：**所有地图事件都转换成一个“牌桌场景”进行交互。**

### **3.1 事件触发流程**
1. 玩家移动到一个“随机事件”节点。
2. 场景平滑切换到 `EventScene.tscn`（布局类似于 [deck.tscn](file:///z:/root/project/godot/carddemo/Scenes/Deck/deck.tscn)）。
3. `EventManager` 根据事件 ID，从 `Deck` 中动态“发牌”给玩家。

### **3.2 交互示例：森林遗迹事件**
当玩家进入该节点，牌桌上会出现三张卡：
- **[卡片 A: 搜索遗迹]**: 效果为“获得 1 张随机道具卡，但触发一场战斗”。
- **[卡片 B: 休息片刻]**: 效果为“回复 10 点 HP，但跳过下一回合”。
- **[卡片 C: 悄悄离开]**: 效果为“直接回到地图场景”。

**交互方式**: 玩家点击或拖拽卡牌来做出选择，完全复用现有的 [Card.cs](file:///z:/root/project/godot/carddemo/Scripts/Cards/Card.cs) 逻辑。

## 4. 技术实现路径

### **4.1 地图场景结构**
- **MapRoot (Node2D)**: 顶层容器。
- **Camera2D**: 支持缩放和平移，查看大地图。
- **NodeLayer**: 存放所有地图节点的容器。
- **PathLayer (Line2D)**: 绘制节点间的连接线。

### **4.2 内存管理建议**
- **对象池 (Object Pooling)**: 地图节点（MapNode）使用对象池，避免频繁切换片段时产生垃圾回收。
- **资源预加载**: 当玩家接近“出口”节点时，在后台预加载下一个片段的资源。

---

## 5. 与现有系统集成

- **GlobalManager**: 添加 `GotoMap()` 和 `GotoEvent(int eventId)` 方法。
- **UIManager**: 地图 UI 作为一个常驻面板，由 `UIManager` 切换显示隐藏。
- **StorageManager**: 在 `PlayerInfo` 中记录 `CurrentMapID` 和 `CurrentNodeID`，实现存档即停、读档即玩。
